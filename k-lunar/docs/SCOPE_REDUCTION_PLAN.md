# 범위 축소 계획

## 목표

`k-lunar`를 넓은 범위의 `lunar-javascript` 포크에서, 지원 범위 전체의 그레고리력 양력 출력을 제공하는 음력/양력 변환 중심 패키지로 줄입니다.

## 목표 범위

패키지는 주로 다음을 지원해야 합니다.

- 양력 -> 음력 변환
- 음력 -> 양력 변환
- 역사 날짜를 포함한 그레고리력 양력 처리
- 위 변환을 지원하는 최소한의 유틸리티

## 기본적으로 범위 밖인 항목

다음 영역은 `k-lunar`의 주목표가 아닙니다.

- 불교력 헬퍼
- 도교력 헬퍼
- 구성(九星) 관련 헬퍼
- 공휴일 유틸리티
- 별자리/황도 12궁 헬퍼
- 복(伏), 수구(数九), 오후(物候), 육요 등 연감 파생 기능
- 사주 및 운세 분석 기능

이 기능들은 당분간 단일형 upstream 런타임 안에 남아 있을 수는 있지만, 공개 API나 기본 테스트 범위를 정의해서는 안 됩니다.

## 축소 전략

### 1단계: 범위 선언

- 변환 전용 목표를 문서화
- 기본 패키지 export를 축소
- 기본 테스트 명령을 변환 관련 테스트로 제한
- 변환 관련 모듈만 노출하는 TypeScript 패키지 엔트리 제공

### 2단계: 코드 분리

- `Solar`, `Lunar`, `SolarUtil`, `LunarUtil`에 실제로 필요한 `src/lunar.ts` 구간 식별
- 변환 핵심 로직과 기타 파생 기능을 분리
- 안전한 범위에서 죽은 export와 불필요한 결합 제거

### 3단계: 런타임 슬림화

- 가능하다면 더 작은 변환 전용 런타임 파일 추출
- 대체 커버리지가 충분해지면 불필요한 테스트와 문서 정리
- 배포 전에 패키지 크기와 공개 API 명확성 검증

## 1단계 산출물

- `src/index.ts`는 변환 중심 API만 export
- `src/`와 `dist/`는 TypeScript 패키징용으로 동일한 변환 중심 API 제공
- `npm test`는 변환 중심 테스트 스위트만 실행
- 관련 없는 upstream 테스트는 `__tests__/`에서 제거

## 주요 위험

`src/lunar.ts`는 여전히 단일형 구조이기 때문에, 겉보기에는 관련 없어 보이는 로직이 실제 변환을 간접 지원할 수 있습니다. 그래서 소스 삭제는 의존 경계가 더 명확해진 뒤에만 진행해야 합니다.
