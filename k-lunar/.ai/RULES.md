# 프로젝트 규칙

## 범위 규칙

- 이 워크스페이스의 일반 작업에서는 `k-lunar/`만 수정합니다.
- `../kasi-lunar/`는 읽기 전용 참고 자료로 취급합니다.

## 변경 규칙

- 큰 재작성보다 작고 검토하기 쉬운 변경을 선호합니다.
- 변경이 의도적이고 문서화된 경우가 아니라면 upstream API 호환성을 유지합니다.
- 달력 정책 변경은 명시적이고 diff로 보기 쉽게 유지합니다.
- 연감, 운세 기능보다 변환 관련 API와 테스트를 우선합니다.
- 동작이 바뀌면 테스트나 하네스 출력 대상도 함께 갱신합니다.
- 패키지 레이아웃을 바꿀 때는 TS 빌드가 충분히 검증될 때까지 패키지 엔트리가 계속 동작해야 합니다.

## 검증 규칙

- 집중 수정에서는 보통 다음 순서를 권장합니다.
  1. `npm run build`
  2. `npm run harness:smoke`
  3. `npm run harness:report`
  4. `npm test`
- Node.js가 없으면 하네스 파일은 최신 상태로 유지하고, 런타임 검증을 실행하지 못했다는 점을 기록합니다.

## 문서 규칙

- 오래 유지할 사실은 `.ai/MEMORY.md`에 둡니다.
- 현재 구현 방향은 `.ai/PLAN.md`에 둡니다.
- 에이전트 작업 지침은 `AGENTS.md`에 둡니다.
- upstream 기준이 바뀌면 `.ai/MEMORY.md`도 함께 갱신합니다.
